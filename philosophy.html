Processo

Como abordo desenvolvimento front-end: decisões conscientes, escopo fechado e entrega com qualidade.

Princípios Fundamentais

1. Escopo Fechado Antes de Código

Não começo a escrever código sem entender o problema e definir limites claros. Isso significa:

  • Identificar o problema central que precisa ser resolvido
  • Definir o que está dentro e fora do escopo
  • Estabelecer critérios de sucesso mensuráveis
  • Documentar decisões antes de implementar

2. Performance é Requisito, Não Feature

Performance não é algo que "adiciono depois". É parte do processo desde o início:

  • Lighthouse rodando em cada etapa importante
  • Animações testadas para manter 60fps
  • Lazy loading e otimização de assets por padrão
  • Core Web Vitals como métrica de qualidade

3. Código Legível > Código "Inteligente"

Priorizo clareza sobre demonstração de técnica. Um código que outro desenvolvedor (ou eu mesmo em 6 meses) consegue entender rapidamente é mais valioso que uma solução "elegante" mas obscura.

  • Nomes de variáveis e funções descritivos
  • Comentários explicando o "porquê", não o "o quê"
  • Estrutura de pastas que comunica intenção
  • Commits semânticos e descritivos

4. Cortar é Tão Importante Quanto Adicionar

A habilidade de dizer "não" a features é crucial. Em cada projeto, documento explicitamente o que foi cortado e por quê:

  • Features que adicionariam complexidade sem valor proporcional
  • Otimizações prematuras que atrasariam entrega
  • Efeitos visuais que comprometeriam performance
  • Funcionalidades que podem ser adicionadas em iterações futuras

Fluxo de Trabalho

  1. Entendimento

    Analiso o problema, pesquiso referências, identifico requisitos técnicos e não-técnicos. Faço perguntas antes de assumir.

  2. Planejamento

    Defino arquitetura, escolho stack com justificativa, estabeleço escopo e crio checklist de entrega. Documento decisões importantes.

  3. Implementação

    Código incremental com commits frequentes. Testo em cada etapa. Mantenho código limpo e organizado desde o início.

  4. Validação

    Testo em diferentes dispositivos e navegadores. Rodo Lighthouse. Verifico acessibilidade básica. Reviso código antes de considerar "pronto".

  5. Documentação

    README claro, comentários onde necessário, decisões técnicas documentadas. O projeto deve ser compreensível sem minha presença.

Filosofia de Stack

Escolho ferramentas baseado no problema, não em hype. Algumas diretrizes:

Vanilla First

Se posso resolver com HTML/CSS/JS puro, não adiciono dependências. Frameworks são ferramentas, não requisitos.

Progressive Enhancement

Funcionalidade básica funciona sem JavaScript. Melhorias progressivas adicionam experiência, não dependência.

Performance Budget

Cada biblioteca adicionada tem custo. Avalio se o benefício justifica o peso no bundle final.

Acessibilidade Desde o Início

HTML semântico, ARIA quando necessário, navegação por teclado, contraste adequado. Não é "extra", é padrão.

Aprendizado Contínuo

Cada projeto é uma oportunidade de aprender. Mantenho um processo de retrospectiva pessoal:

  • O que funcionou bem e por quê?
  • Quais decisões eu mudaria com o conhecimento atual?
  • Que padrões ou técnicas novas aprendi?
  • Como posso aplicar esses aprendizados no próximo projeto?

Essa reflexão está documentada em cada case study que apresento.
