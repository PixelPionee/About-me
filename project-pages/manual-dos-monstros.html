Manual de Campo do Caçador

Bestiário interativo com foco em leitura e SEO

Visão Geral

Projeto autoral de bestiário interativo focado em leitura, narrativa e arquitetura estática. O objetivo é criar um sistema de conteúdo que priorize SEO, performance e manutenibilidade, separando completamente dados da interface.

Status: Projeto em desenvolvimento ativo. A arquitetura base está definida e implementada, com foco atual na criação de conteúdo e refinamento da experiência de leitura.

O Problema

Como criar um sistema de conteúdo escalável que priorize leitura, SEO e manutenibilidade?

Bestiários e wikis de jogos frequentemente sofrem de problemas comuns:

  • Conteúdo difícil de manter e atualizar
  • SEO comprometido por renderização client-side
  • Performance ruim devido a excesso de JavaScript
  • Experiência de leitura prejudicada por distrações visuais

O desafio era criar uma solução que resolvesse esses problemas mantendo flexibilidade para crescimento futuro.

A Decisão Principal

Static Site Generation com dados em JSON.

Cada criatura é uma página estática gerada em build time. Isso garante:

  • 100% do conteúdo indexável por motores de busca
  • Performance máxima (páginas pré-renderizadas)
  • Separação clara entre dados e apresentação
  • Facilidade de manutenção (editar JSON é mais simples que editar HTML)

A escolha de Next.js com SSG foi estratégica: oferece as vantagens de um framework moderno mantendo a simplicidade de sites estáticos.

Stack Técnica

Next.js

Framework com suporte nativo a SSG e otimizações automáticas

TypeScript

Type safety para dados estruturados e prevenção de erros

JSON

Formato de dados simples e fácil de manter

Arquitetura de Dados

Cada criatura é definida em um arquivo JSON com estrutura padronizada: 

{
  "id": "criatura-exemplo",
  "name": "Nome da Criatura",
  "category": "Categoria",
  "habitat": "Descrição do habitat",
  "behavior": "Padrões de comportamento",
  "weaknesses": ["Fraqueza 1", "Fraqueza 2"],
  "lore": "História e contexto narrativo"
}

Métricas de Qualidade

  • 100% conteúdo estático indexável por motores de busca
  • Lighthouse Score 95+ em todas as métricas
  • Separação total dados/interface para manutenibilidade
  • Type safety com TypeScript em toda a aplicação

O Que Foi Cortado (E Por Quê)

Decisões de escopo focadas em MVP funcional e qualidade de leitura:

Sistema de Busca Client-Side

Motivo: Adicionaria complexidade prematura. Com poucas criaturas inicialmente, navegação por categorias é suficiente. Busca pode ser adicionada quando o conteúdo crescer.

Animações de Entrada

Motivo: Distração da leitura. O foco é conteúdo, não efeitos visuais. Animações sutis podem ser adicionadas depois, mas nunca devem competir com o texto.

Modo Escuro

Motivo: Foco em MVP funcional. Melhor ter um tema claro bem executado que dois temas medianos. Modo escuro será implementado após validação do design principal.

Sistema de Comentários

Motivo: Escopo e manutenção. Adicionar interatividade social exigiria backend, moderação e complexidade de infraestrutura desproporcional ao valor inicial.

Aprendizados

  • SSG resolve 80% dos problemas de performance e SEO: Páginas estáticas são rápidas, indexáveis e fáceis de hospedar. A complexidade de SPA só se justifica quando há necessidade real de interatividade dinâmica.
  • Separar dados da UI facilita manutenção e testes: Editar conteúdo em JSON é mais simples e menos propenso a erros que editar diretamente em componentes. Também permite validação automática de estrutura.
  • Arquitetura clara > features impressionantes: Um sistema bem organizado que faz poucas coisas bem é mais valioso que um sistema complexo com muitas features mal implementadas.
  • TypeScript previne erros antes de acontecerem: Type safety em dados estruturados evita bugs silenciosos e melhora a experiência de desenvolvimento com autocomplete e validação.
  • Conteúdo primeiro, features depois: É melhor ter conteúdo de qualidade em uma interface simples que uma interface complexa sem conteúdo. O valor está na informação, não na tecnologia.

Próximos Passos

Roadmap de desenvolvimento:

  1. Completar conteúdo inicial (10-15 criaturas)
  2. Implementar sistema de categorias e filtros
  3. Adicionar modo escuro após validação do design principal
  4. Otimizar imagens e implementar lazy loading avançado
  5. Considerar busca client-side quando conteúdo atingir massa crítica
