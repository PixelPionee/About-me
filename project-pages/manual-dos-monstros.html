Manual de Campo do Caçador

Bestiário digital imersivo baseado na lore literária de The Witcher

Visão Geral

Uma aplicação de arquitetura de informação focada em UI Diegética (interface que existe dentro do universo narrativo). O projeto simula um bestiário físico antigo, priorizando a imersão do leitor sobre padrões convencionais de navegação web.

O Problema de UX

A maioria das Wikis e bases de dados de jogos quebra a imersão com layouts genéricos, anúncios e excesso de elementos de UI.

O desafio foi arquitetar uma solução que parecesse um "livro físico envelhecido" na tela, mas que por trás rodasse como uma aplicação moderna (Next.js).

Isso significa garantir indexação (SEO), carregamento instantâneo e acessibilidade — algo que sites feitos inteiramente em Canvas/Flash não conseguem entregar.

A Decisão Principal

SSG com Next.js 14 + Dados em JSON Local + Estética de Manual Envelhecido

Cada criatura é uma página estática gerada em build time, com dados estruturados em JSON que separam completamente conteúdo da apresentação. Isso garante:

  • 100% do conteúdo indexável por motores de busca
  • Performance máxima (páginas pré-renderizadas)
  • Fidelidade literária (dados baseados nos livros de Sapkowski)
  • Facilidade de manutenção (editar JSON é mais simples que editar HTML)

Stack Técnica

Next.js 14+

App Router com SSG nativo

TypeScript

Type safety para dados estruturados

CSS Modules

Estilização sem Tailwind

JSON Local

Sem banco de dados

Estrutura de Dados

Todas as criaturas estão em data/bestiario.json com a seguinte estrutura: 

interface Criatura {
  id: string;
  nome_comum: string;
  classificacao: string;
  ilustracao_src: string;
  habitat: string;
  comportamento: string;
  nivel_ameaca: string;
  observacoes_cacador: string;
}

Métricas de Qualidade

  • 100% conteúdo estático indexável por motores de busca
  • Fidelidade literária às descrições de Sapkowski
  • Separação total dados/interface para manutenibilidade
  • Type safety com TypeScript em toda a aplicação

O Que Foi Cortado (Decisões de Escopo)

Demonstração de pensamento estratégico através de trade-offs conscientes:

Barra de Busca (Search Input)

Planejado: Sistema de busca global com fuzzy search para filtrar criaturas
Implementado: Índice Visual e Filtros por Categoria (Abas)
Por quê: Para um MVP com 12 criaturas, a busca textual é overengineering. O índice visual é mais rápido e mantém a imersão de "folhear um livro", enquanto a busca faria parecer uma Wiki genérica.

Modo Escuro (Dark Mode)

Planejado: Toggle de tema Claro/Escuro conforme preferência do sistema
Implementado: Tema Único (Papel Envelhecido)
Por quê: Decisão Artística e Técnica. O "Dark Mode" inverteria as cores do papel, quebrando a ilusão de que o site é um objeto físico antigo. A consistência atmosférica vale mais que a feature padrão.

Aprendizados

  • SSG resolve 80% dos problemas de performance e SEO: Páginas estáticas são rápidas, indexáveis e fáceis de hospedar. A complexidade de SPA só se justifica quando há necessidade real de interatividade dinâmica.
  • Separar dados da UI facilita manutenção: Editar conteúdo em JSON é mais simples e menos propenso a erros que editar diretamente em componentes. Também permite validação automática de estrutura.
  • Fidelidade literária > interpretações visuais populares: Seguir os livros de Sapkowski, não os jogos, cria um produto único e autêntico.
  • Conteúdo primeiro, features depois: É melhor ter conteúdo de qualidade em uma interface simples que uma interface complexa sem conteúdo. O valor está na informação, não na tecnologia.
