Ghost of Tsushima — Interactive Experience

Narrativa visual através de código e performance

Visão Geral

Projeto pessoal focado em traduzir a atmosfera contemplativa de Ghost of Tsushima para uma experiência web interativa. O objetivo era criar uma narrativa visual através de código, mantendo performance impecável e fluidez nas animações.

O Problema

Como traduzir a atmosfera contemplativa de um jogo em uma experiência web interativa sem comprometer performance?

Ghost of Tsushima é conhecido por sua direção de arte minimalista e momentos contemplativos. O desafio era capturar essa essência usando tecnologias web, mantendo 60fps constantes e criando uma experiência que não parecesse "apenas um site".

A Decisão Principal

Priorizar fluidez visual sobre quantidade de efeitos.

Cada partícula foi otimizada. Animações desnecessárias foram cortadas. O foco estava em criar poucos elementos visuais, mas executados perfeitamente.

Isso significou usar Canvas API de forma estratégica, implementar pooling de objetos para partículas e limitar efeitos simultâneos baseado em testes de performance em diferentes dispositivos.

Stack Técnica

React

Gerenciamento de estado e componentes reutilizáveis

Canvas API

Sistema de partículas e animações de baixo nível

JavaScript ES6+

Lógica de animação e otimizações de performance

Métricas de Performance

  • 60fps constantes em animações e interações
  • Sistema de partículas otimizado com object pooling
  • Lazy loading de assets visuais
  • Throttling em event listeners para scroll e mouse

O Que Foi Cortado (E Por Quê)

Decisões conscientes de escopo para garantir qualidade sobre quantidade:

Sistema de Clima Dinâmico

Motivo: Complexidade vs. valor. Adicionar chuva, neve e vento dinâmicos exigiria muito mais processamento sem contribuir significativamente para a experiência central.

Trilha Sonora Adaptativa

Motivo: Escopo técnico. Implementar áudio que responde a interações do usuário adicionaria complexidade de sincronização e questões de autoplay em navegadores.

Modo Noturno Animado

Motivo: Performance. Transições de dia/noite com re-renderização de partículas e mudanças de iluminação comprometeriam os 60fps em dispositivos médios.

Aprendizados

  • Canvas API exige trade-offs conscientes entre visual e CPU: Cada partícula adicional tem custo. É melhor ter 50 partículas fluidas que 200 travando.
  • 60fps não é negociável em experiências interativas: Usuários percebem imediatamente quando animações não são suaves. Performance é parte da experiência, não um "extra".
  • Menos efeitos bem executados > muitos efeitos medianos: Minimalismo intencional comunica mais maturidade que excesso de features.
  • Object pooling é essencial para partículas: Criar e destruir objetos constantemente causa garbage collection. Reutilizar objetos mantém performance estável.
